Project Summary

This project is an application development for a (fictitious) burger restaurant in Wellington which aims to allow customers to create a custom burger in a user-friendly manner. We have implemented a database to store the orders, burger ingredients, and a list of customers connected with the production line application using Java which the staff at Dimmu Burger will use. The completion of this project would benefit Dimmu Burger and its users because of a user-friendly interface that allows customers to order burgers.

Integration

Heroku

Heroku is a cloud platform for hosting web applications and databases. We chose to use Heroku because we thought that it would be a good way for everyone to access everything from anywhere. We looked into a few other hosting options at the time such as Google FireBase, with which you can also host web applications for free, but ultimately chose to use Heroku because we also wanted to use a SQL database with a schema.

To get started with Heroku we followed the documentation explaining how to host a PHP project with Heroku and provision a Heroku PostGreSQL database to it. It went through installing PHP and a dependency manager for PHP called Composer which allows you to declare PHP libraries you will be using in your project.

PHP + PDO

In our Heroku web app, we used PDO or PHP Data Objects rather than SQLi to make calls and connect to the database. This came as we soon realised that we were developing with both a MariaDB database and a PostGres Database. Looking at our options, we could have either changed it to PostGres or PDO - which has the benefit of having drivers available for many different SQL databases and also seemed to be better regarding security and the risk of SQL injection as PDO uses prepared statements when connecting to the database. Overall, though we would need more time to look into different security risks to say that our website is truly secure.

AJAX

One challenging thing that we wanted to do was to display when the status of a burger changes from ‘new’ to ‘received’ to ‘in progress’ to ‘complete’ as it was updated by the kitchen staff in the Production Line app. Originally we had the idea that we might be able to ask the database if there had been any changes to the database in a timed interval, say every 3 seconds. AJAX, which is a set of techniques in Javascript uses HTTP request and allows the web page to be asynchronously updated. To get this set up, we mostly followed this guide https://www.w3schools.com/xml/ajax_intro.asp as AJAX has a set way of doing things such as check the status is 200 (everything O.K.)

Reflection

What We Changed From Our Original Plan

  • Ordering multiple burgers within a single order.
    • We initially planned for an order to be able to contain multiple different burgers within it. However, as we approached the end of the project and this feature had not been implemented, a decision was made to leave this option out.
  • The customer gets an estimated time until order ready.
    • Once the customer had placed an order, there was going to be a feature on the final page (that displayed their order number and order state) that gave an estimated time until their order was complete. This would have been based on how many burgers were in the queue in front of theirs. This feature was not implemented due to time constraints.
  • Customer orders not pulling stock levels from the kitchen (can order out of stock ingredients)
    • The web application was meant to make a stock level request from the database before it loaded the webpage, and only display the ingredients that were currently in stock. This would be to prevent from ordering ingredients that were currently unavailable. Unfortunately, the team lost sight of this requirement while working on the application, and when we realised it was missing, we did not have enough time left to implement it.
  • Originally, the webpage also talked to jdbc class
    • When we initially designed the application structure, it was envisioned that the webpage would also be talking to the jdbc class, and the jdbc class would be the only class talking to the database. As the project progressed, we realised that our web application worked better if it was talking directly to the database. This still left the database as the only link between the front- and back-end applications, which was desired.
